# 前端调试预览工作流

## 目标

前端功能开发完成后，不直接停留在“代码改完”或“测试通过”，而是进入可预览、可复现、可定位问题的调试验收流程。

## 默认流程

1. 先确认问题或需求的真实表现，优先定位根因，不先猜改法。
2. 先补失败测试，再做最小修复，避免一边改一边猜。
3. 修复后重新运行相关测试、类型检查；必要时补充构建验证。
4. 启动 `frontend` 本地 dev 服务，默认使用可直接访问的地址，例如：
   - `http://127.0.0.1:5173/images`
   - `http://127.0.0.1:5173/...`
5. 使用浏览器进行实测预览，检查：
   - 布局宽度、边距、对齐
   - 初始化状态是否闪烁
   - 交互流程是否符合预期
   - 文案、按钮状态、异常提示是否合理
6. 调试完成后，把可访问地址提供给用户，方便用户立即手动复核。

## 执行原则

- 优先遵循：定位根因 → 失败测试 → 最小修复 → 测试/类型检查 → 浏览器实测。
- 浏览器实测不是可选项，而是前端改动的默认验收方式。
- 如果用户继续反馈视觉或交互问题，优先基于本地 dev 预览继续迭代，而不是只靠静态代码判断。
- 若临时启动了本地 dev 服务，完成后可按需关闭，但在用户仍需预览时应保持可访问。
