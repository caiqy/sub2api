# 上游更新合并工作流

## 目标

在 `sub2api` 仓库中合并上游更新时，优先保证三件事：

1. 找准真正需要合并的上游目标（分支 vs release tag）。
2. 在不丢失本地定制能力的前提下吸收上游修复和功能。
3. 合并后不仅看“能编译/测试通过”，还要专项审查本地关键行为是否被上游语义性覆盖。

## 默认流程

1. 先检查当前分支、工作区状态、`HEAD` 是否适合承接合并。
2. 获取 upstream 的分支和 tags，不预设 `upstream/main` 一定代表最新发布状态。
3. 对比：
   - 本地当前版本/tag
   - upstream 最新 tag
   - `upstream/main` 与本地主线的祖先关系
4. 如果 `upstream/main` 已经是本地主线祖先、但 upstream 有更新 release tag，则优先按目标 release tag 合并，而不是误以为“已经同步完了”。
5. 默认先从本地主线切出临时分支，在临时分支上完成合并。
6. 冲突处理时，优先采用“保留上游变更 + 保留本地定制”的合并方式；只有在真实业务语义冲突、无法同时成立时，再整理选项给用户确认。
7. 合并完成后，先做针对性 review，再做测试/类型检查/构建验证。
8. 验证通过后，再决定是否合回主分支、是否推送远端。

## 冲突处理原则

- 不要机械地偏向 ours/theirs。
- 先识别冲突是：
  - 上游修复
  - 本地定制
  - 接口/配置演进
  - 纯版本或生成文件差异
- 若两边能力可共存，优先保留两边。
- 若某处是本地专门增强点，合并后必须继续确认入口、条件判断、回退路径是否仍然生效。
- `VERSION`、生成文件、配置项等“容易看起来小但会影响发布/运行”的文件，要单独复核。

## 本次合并得到的关键经验

### 1. 不要默认拿 `upstream/main` 当唯一基准

这次实际有效更新主要在 upstream release tag（如 `v0.1.119`），而不是简单表现为一个可直接 merge 的 `upstream/main` 新分叉。

### 2. 临时分支合并比直接进 `main` 更稳

这样即使后续发现语义问题，也能在隔离环境里继续补测、修复和复审，不污染主线。

### 3. “测试通过”不等于“本地能力没被改坏”

这次第一次合并后验证已通过，但后续重新独立审查，还是发现了 scheduler / sticky / privacy / image capability / 热更新 等语义问题。
因此对本地核心定制点，必须做“是否被上游改写行为”的专项 review。

### 4. 对复杂调度/网关逻辑，优先走失败测试驱动修复

当问题涉及 scheduler、sticky、fallback、runtime config、capability filter 等复杂路径时，先补失败测试，再做最小修复，明显比直接猜改法更稳。

### 5. 重点关注“边界路径”而不是只看主路径

这次后续问题主要出现在：
- previous response sticky
- session sticky
- fallback / WaitPlan
- DB recheck
- 运行时配置热更新后的重建/重载

这些路径最容易在 merge 后被静默破坏。

### 6. 合并后的 review 要围绕“本地关键能力清单”做

例如：
- 本地 scheduler 策略
- sticky 语义
- privacy 要求
- image capability 过滤
- runtime setting 热更新
- 网关透传字段
- 其他本地增强功能

不要只按文件 diff 粗看，要按“能力是否仍成立”逐项检查。

## 建议的合并后验证

至少执行：

- `go test ./...`
- 前端相关定向测试
- `pnpm typecheck`
- `pnpm build`

如果改动涉及前端功能，按既有偏好继续做本地 dev 预览和浏览器实测。

## 执行原则

- 先确认真正的上游目标，再合并。
- 先在临时分支做，别直接动主线。
- 冲突优先保留两边能力，不轻易二选一。
- 测试通过后仍要做语义 review。
- 对复杂行为问题，优先失败测试 + 最小修复。
