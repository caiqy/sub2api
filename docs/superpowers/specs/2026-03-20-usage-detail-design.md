# 管理员使用记录请求/响应详情设计

## 背景

当前项目已经有管理员使用记录页面 `frontend/src/views/admin/UsageView.vue`，可以按用户、模型、时间等条件筛选和查看 usage 列表、统计卡片与图表。但现有 `usage log` 数据结构只承载摘要型信息，例如用户、模型、请求 ID、Token、费用、耗时等，不包含完整的请求/响应原文。

这导致管理员在排查具体一次调用时，无法直接在“查看和管理所有用户的使用记录”界面内看到：

- 请求 header
- 请求 body
- 响应 header
- 响应 body

本设计目标是在不破坏现有 usage 列表、统计、导出与筛选逻辑的前提下，为管理员增加“按需查看原始请求/响应详情”的能力。

## 已确认需求

- 增强管理员“使用记录”页面，而不是新开独立监控页面
- 列表中增加一个入口按钮，点击后弹窗展示详情
- 详情内容包含：
  - request headers
  - request body
  - response headers
  - response body
- 记录范围覆盖成功和失败请求
- 详情按**全局范围**只保留最近 500 条
- 内容按用户要求采用**完整原文存储**
- 弹窗交互采用 **Tabs**，而不是四个普通按钮或双栏对照

## 非目标

- 不重做现有管理员 usage 列表、图表与筛选体系
- 不把该能力并入现有 ops 错误排障视图
- 不在本次设计中引入脱敏、截断、摘要化等安全/容量保护策略（尽管这在长期上存在风险）
- 不要求删除 usage 主记录本身；“最近 500 条”仅作用于详情快照层

## 设计概览

推荐方案是：在现有 usage 主记录之外，新增一层独立的“详情快照”存储。管理员 usage 列表继续走现有轻量接口；当管理员点击某一行的“详情”按钮时，前端再按需请求这条 usage 的详情快照，并通过弹窗中的 4 个 Tabs 展示四块原始内容。

这样可以把“列表查询/统计”与“原始长文本详情”解耦，避免把大字段直接塞入 usage 主表，降低对现有页面性能、分页和导出逻辑的影响。

## 架构与组件边界

### 1. Usage 主记录层（保留现状）

现有 usage 记录继续负责：

- 列表分页
- 筛选与查询
- 聚合统计
- 图表与导出

该层不直接承载大文本详情字段。

### 2. Usage 详情快照层（新增）

新增一张与 usage 记录一对一关联的详情表，例如 `usage_log_details`。其职责仅为：

- 存储请求/响应原文快照
- 支撑管理员查看单条 usage 的详情弹窗
- 通过独立保留策略维持“全局最近 500 条”

### 3. 管理员详情接口层（新增）

新增管理员专用接口，按 usage 记录 ID 返回这条记录的详情快照。

### 4. 管理员前端详情弹窗（新增）

在 `UsageTable` 行级操作中增加“详情”按钮；点击后加载并展示详情弹窗。弹窗采用 Tabs 形式，四个分区分别展示 request/response 的 headers/body。

## 数据模型设计

建议新增表：`usage_log_details`

建议字段：

- `id`
- `usage_log_id`：关联 usage 主记录，唯一
- `request_headers`：文本
- `request_body`：长文本
- `response_headers`：文本
- `response_body`：长文本
- `created_at`

### 约束与索引

- `usage_log_id` 唯一索引：保证一条 usage 记录仅对应一份详情快照
- `created_at` 索引：支持按新旧顺序清理，仅保留最近 500 条
- 可选外键/级联删除：如果 usage 主记录未来被清理，详情记录应能一起移除，避免孤儿数据

### 关键决策

“最近 500 条”的裁剪目标是**详情表**而不是 usage 主表。这意味着：

- usage 主记录仍保持现有历史能力
- 只有详情查看窗口受 500 条限制
- 当列表中较旧的 usage 记录没有对应详情时，前端需要给出明确提示

## 接口设计

### 新增接口

`GET /api/v1/admin/usage/:id/detail`

返回建议结构：

```json
{
  "usage_log_id": 123,
  "request_headers": "...",
  "request_body": "...",
  "response_headers": "...",
  "response_body": "...",
  "created_at": "2026-03-20T10:21:33Z"
}
```

### 错误语义

- usage 主记录不存在：返回 404
- usage 主记录存在，但详情快照不存在：也返回 404，错误消息应明确表示“详情不在最近 500 条保留窗口内”或“详情不存在”
- 当前用户不是管理员：返回 403

### 列表接口增强

列表接口不返回四个大字段，但建议增加一个轻量字段：

- `has_detail: boolean`

目的：

- 前端可直接判断是否显示可点击的“详情”按钮
- 对于较旧记录，可提前展示不可用状态而不是点开后才失败

若实现复杂度较高，也可以在首版省略 `has_detail`，统一显示按钮，点击时再查询详情接口并处理 404。但从交互完整性看，推荐增加该字段。

## 数据写入与保留策略

### 写入流程

每次请求完成并生成 usage 主记录后：

1. 创建 usage 主记录
2. 采集并写入该次请求的 request/response 详情快照
3. 快照写入成功后，执行保留策略
4. 删除超出保留窗口之外的旧详情快照

### 保留策略

按全局范围排序：

- 以 `created_at desc, id desc` 为准
- 保留前 500 条详情快照
- 删除其余更旧的详情快照

### 一致性预期

- 允许 usage 主记录存在但详情快照不存在
- 不要求详情快照成为统计或导出的数据来源
- 列表和图表不应依赖详情快照是否存在

## 前端交互设计

### 列表交互

在管理员 usage 列表的每一行新增“详情”按钮，放在现有行操作区域。

交互行为：

- 有详情：点击后打开弹窗并加载内容
- 无详情：按钮禁用或显示不可用状态；若未实现 `has_detail`，则在点击后以错误态提示

### 弹窗结构

使用居中详情弹窗，而不是右侧抽屉或双栏对照。

弹窗头部展示：

- request id
- 用户标识（例如邮箱）
- 模型名
- 请求时间
- “仅保留全局最近 500 条详情”的辅助提示

弹窗主体使用 4 个标准 Tabs：

1. `Request Headers`
2. `Request Body`
3. `Response Headers`
4. `Response Body`

每个 Tab 使用单独的大文本区域展示完整内容，保留换行与格式。

### 弹窗操作

建议支持：

- 复制当前 Tab 内容
- 关闭弹窗
- 对 JSON 文本做尽力格式化显示；若不是合法 JSON，则按原文显示

### 空状态与错误态

- 详情不存在：显示“该记录详情已不在最近 500 条保留窗口内”
- 加载失败：显示通用错误态，并允许重试
- 内容为空：显示明确的“无内容”占位，而不是空白区域

## 后端实现边界

### 不复用 ops 详情作为主承载

现有 ops 体系已经对错误场景捕获了部分 request body/header 信息，但其目标是运维排障，而不是管理员 usage 的全量成功/失败请求详情展示。

因此本设计不建议直接复用 ops 作为主数据源，原因包括：

- 语义不匹配：ops 偏错误与重试排障，usage 详情偏审计与查看
- 覆盖范围不匹配：本需求要求成功和失败请求都记录
- 数据边界不清晰：继续扩展会让 admin usage 与 ops 的职责混淆

可以复用局部捕获能力或工具函数，但最终数据模型与接口应独立归属 usage detail。

## 风险与取舍

### 1. 完整原文存储的安全风险

本次需求已明确选择“完整原文存储”，因此设计不做脱敏和截断。但这意味着：

- 敏感 header（如 Authorization、Cookie）可能被管理员直接看到
- body 内可能包含用户输入、上游返回的敏感内容或内部字段

这是一个明确接受的产品取舍，应在实现和上线说明中再次强调。

### 2. 大文本存储与性能风险

完整保存 request/response body 可能带来较大的数据库写入与存储压力，尤其是长 prompt、长 response 或流式聚合内容。之所以仍可接受，是因为：

- 明细独立存储，不拖慢 usage 列表主查询
- 全局只保留最近 500 条，容量有硬上限

### 3. 详情缺失的用户体验

由于只保留最近 500 条详情，管理员查看较旧 usage 时可能无法打开原文详情。这不是异常，而是设计预期，因此前端必须给出清晰提示，避免用户误以为系统出错。

## 测试设计

### 后端测试

- 详情表读写测试
- 保留策略测试：新增第 501 条后，最旧详情被清理
- 管理员详情接口测试：
  - 有详情时返回 200
  - usage 不存在时返回 404
  - detail 不存在时返回 404
  - 非管理员访问返回 403
- `has_detail` 字段映射测试（若实现）

### 前端测试

- 列表“详情”按钮渲染测试
- 点击按钮后请求详情接口并打开弹窗
- Tabs 切换测试
- JSON/纯文本展示测试
- “详情不存在”错误态展示测试
- 复制当前 Tab 内容交互测试

### 集成验证

- 在管理员 usage 页面筛选、分页、导出功能不受影响
- 打开详情弹窗不会阻塞列表刷新与翻页
- 老记录在无详情时表现稳定、提示明确

## 验收标准

满足以下条件即可视为功能完成：

1. 管理员 usage 列表每条记录具备“详情”查看入口
2. 点击后可在弹窗中查看 request headers/body 与 response headers/body
3. 弹窗采用标准 Tabs 交互
4. 详情只保留全局最近 500 条
5. 较旧记录无详情时有明确提示
6. 现有 usage 列表、统计、图表、导出能力不被破坏
